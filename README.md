# LMS-система — учебный проект на Django и DRF

## Описание

Это учебный проект системы управления обучением (LMS), разрабатываемый поэтапно. На каждом этапе реализуются новые
компоненты: от базовой настройки окружения до построения полноценного REST API для работы с курсами, уроками,
пользователями и платежами. К завершению проекта будет создан прототип LMS-платформы, пригодный для портфолио и
дальнейшего развития.

## Цели проекта

- Освоить разработку REST API на Django и Django REST Framework.
- Отработать практику работы с моделями, сериализаторами, вьюсетами и фильтрами.
- Научиться структурировать проект по приложениям.
- Реализовать полноценную CRUD-логику для основных сущностей.
- Получить опыт тестирования API с помощью Postman.

## Текущий статус

**Этап разработки:** Реализованы основные модели, API, сериализация, фильтрация, управление доступом, валидация,
подписки, пагинация, тесты, документация и интеграция оплаты.

---

## Этап 1: Базовая настройка и реализация основных сущностей

### Реализовано

- Создан новый Django-проект, подключен Django REST Framework (DRF).
- Реализовано приложение `users` с моделью пользователя:
    - Авторизация по email.
    - Дополнительные поля: телефон, город, аватарка.
- Создано приложение `lms` (или `materials`) с моделями:
    - **Курс**: название, превью, описание.
    - **Урок**: название, описание, превью, ссылка на видео.
    - Реализована связь «один ко многим» между курсом и уроками.
- Описаны CRUD-операции:
    - Для курса — через Viewsets.
    - Для урока — через Generic-классы.
- Реализованы простейшие сериализаторы для всех моделей.
- Проверена работа всех эндпоинтов с помощью Postman.

> ⚙️ На этом этапе заложен фундамент проекта: реализованы основные сущности и базовые операции, что позволяет начать
> работу с курсами и уроками.

---

## Этап 2: Расширение функционала и оптимизация API

### Реализовано

- В сериализатор курса добавлено поле количества уроков через `SerializerMethodField`.
- В приложение `users` добавлена модель **Платежи**:
    - Пользователь (ForeignKey).
    - Дата оплаты.
    - Оплаченный курс или урок (ForeignKey).
    - Сумма оплаты.
    - Способ оплаты (наличные/перевод).
- В сериализатор курса реализовано поле вывода всех уроков курса (вложенная сериализация).
- Один сериализатор курса теперь возвращает и количество уроков, и список уроков.
- Настроена фильтрация для эндпоинта списка платежей:
    - Сортировка по дате оплаты.
    - Фильтрация по курсу/уроку.
    - Фильтрация по способу оплаты.
- Дополнительное задание: в сериализатор пользователя добавлено поле истории платежей пользователя.

> ⚙️ На этом этапе проект получил расширенную аналитику по курсам и удобную систему учёта платежей, что делает его более
> функциональным для пользователей и администраторов.

## Этап 3: Управление доступом и авторизация

### Реализовано

- **Реализована CRUD-логика для пользователей**, включая регистрацию новых пользователей.
- **Настроена JWT-авторизация** в проекте. Все эндпоинты, кроме регистрации и авторизации, закрыты для неавторизованных
  пользователей.
- **Создана группа «Модераторы»** с ограниченными правами: модераторы могут просматривать и редактировать любые курсы и
  уроки, но не могут их создавать или удалять.
- **Реализован кастомный permission-класс** для проверки принадлежности пользователя к группе модераторов.
- **Группы пользователей** можно назначать через админ-панель. Для автоматизации создана фикстура или кастомная команда
  для заполнения групп в базе данных.
- **Разграничены права доступа**:
    - Для ViewSet-права определяются динамически через метод `get_permissions()`.
    - Для Generic-классов права задаются через атрибут `permission_classes`.
- **Реализована объектная авторизация**: пользователи могут работать только со своими объектами.

> ⚙️ На этом этапе проект получил надёжную систему разграничения прав доступа и авторизации, что обеспечивает
> безопасность данных и гибкое управление ролями пользователей.

## Этап 4: Валидаторы, подписки, пагинация и тесты

### Реализовано

- **Валидатор ссылок на видео**: разрешены только ссылки на *youtube.com*. Валидатор интегрирован в сериализатор урока.
- **Модель подписки на курс**: реализована модель с полями «пользователь» и «курс». Добавлен эндпоинт для управления
  подпиской (добавление/удаление). В сериализатор курса добавлено поле статуса подписки пользователя.
- **Пагинация для курсов и уроков**: реализованы классы пагинации с параметрами *page_size*, *page_size_query_param*,
  *max_page_size*. Пагинатор подключён к контроллерам.
- **Тестирование**: написаны тесты для CRUD уроков и функционала подписки. Использован метод `setUp` для подготовки
  данных. Проверены сценарии с разными правами доступа.

> ⚙️ На этом этапе проект стал более удобным для пользователя: добавлены подписки на курсы, реализована пагинация для
> больших списков и обеспечена стабильность кода за счёт автотестов.

---

## Этап 5: Документация и интеграция оплаты через Stripe

### Реализовано

- **Автоматическая документация API**: подключена библиотека *drf-yasg* или *drf-spectacular*. Каждый эндпоинт описан в
  документации. Документация доступна по адресу */swagger/* или */redoc/*.
- **Интеграция оплаты через Stripe**:
    - Реализована логика взаимодействия с *Stripe API* (создание продукта, цены, *checkout*-сессии).
    - В сервисных функциях реализовано создание продукта и цены в *Stripe*, формирование *checkout*-сессии с передачей
      *id* цены, сохранение ссылки на оплату в модели платежа.
    - В ответе на POST-запрос пользователь получает ссылку на оплату или данные о платеже.
    - Для тестирования используются тестовые карты из документации *Stripe*.

> ⚙️ На этом этапе проект получил полноценную документацию для разработчиков и пользователей, а также готовую интеграцию
> с платёжной системой, что делает его готовым к демонстрации и дальнейшему развитию.

| **Технология**                   | **Назначение**                                |

| **Backend**                      | Django, Django REST Framework (DRF)           |

- *Redis*: используется как брокер или кеш (в зависимости от конфигурации).

- *Coverage*: для анализа покрытия кода тестами.

## Тестирование API

Все эндпоинты протестированы с помощью Postman. Примеры запросов:

- **GET** `/api/courses/` — список курсов
- **POST** `/api/lessons/` — создание урока
- **GET** `/api/payments/?course=1&ordering=-date` — фильтрация платежей по курсу и сортировка по дате
